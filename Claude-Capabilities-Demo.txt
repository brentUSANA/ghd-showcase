  1. JIRA TICKET AUTOMATION  (The Big One)
--------------------------------------------------------------------------------

Claude can create, close, and comment on GHD tickets in Jira from plain English
with zero copy-paste into a browser.

WHAT IT DOES AUTOMATICALLY:
  - Sets the correct Component (Printer, Software, VPN, AD, Cell Phone, etc.)
    based on keywords in the description
  - Sets Sub-Component (Hardware, Software - Issues, Software - Install, etc.)
  - Detects the Channel (Phone Call, Walk Up, Chat, Email, Ticket)
  - Sets the correct Reporter to the affected user (not Brent but will fall back on Brent of no user located)
  - Copies the ticket URL to the clipboard the moment it's created
  - Includes Mac devices via Jamf (WINOPS-19019 closed April 2026 -- now live)
  - Closes the ticket immediately if resolved (FCR = Yes if applicable, status = Closed)
  - Posts a separate resolution comment so the description stays clean

EXAMPLE INTERACTIONS:

    weren't showing up. I switched her to the Local PC Recording tab and
    it was resolved."
  --> Claude creates GHD-97384, sets Component: Software, Sub-Component:
      Software - Issues, Channel: Phone Call, FCR: Yes, Reporter: Carolyn Green,
      closes it, and copies the URL to your clipboard to paste in to browser easily.
      https://usana.atlassian.net/browse/GHD-97384

  > "Walk-up -- Charlotte Mukayiranga brought her PC in, can't log into Agile PLM."
  --> Claude sets Component: Software, Sub-Component: Software - Issues,
      Channel: Walk Up, Reporter: Charlotte Mukayiranga. Knows Agile uses its
      own credentials (not SSO), walks through the fix sequence.
      https://usana.atlassian.net/browse/GHD-97387

  > "AD reset ticket for Hatidza -- Brayden in HR called for her."
  --> Claude creates the ticket with Brayden as the caller context noted,
      Hatidza as the reporter, FCR, closed. One command.

  > "Leave the ticket open -- still waiting on user to test."

PIM ROLE ACTIVATION -- AUTOMATED (/pim COMMAND):
  Many AD, Exchange, and Azure operations require activating a PIM role first.
  Previously this meant: portal.azure.com -> PIM -> select role -> set duration
  -> justify -> submit -> wait. Done in a browser, done manually, every time.

  Now: type /pim in the Claude Code terminal. A numbered menu appears with every
  available role (Helpdesk Admin, User Admin, Cloud Device Admin, Exchange Admin,
  etc.). Pick one, enter duration and justification. The slash command triggers a
  PowerShell script that activates the role via the Microsoft Graph API.
  No browser. No portal navigation. Done.

  Auth is handled via device code flow on first use -- token is cached for the
  session. Works silently for all subsequent activations in the same session.

  PROACTIVE AUTH CHECK -- MORNING DIGEST:
  The delegated Graph token (used by /pim) expires overnight. To prevent this
  from surfacing mid-interaction -- e.g., a walk-up user waiting while a device
  code prompt appears -- Claude runs a Graph auth check at the start of every
  morning digest. If the token is still valid, it notes "PIM auth OK" and moves
  on. If it's expired, it immediately triggers the device code login and presents
  the URL + code at the top of the digest, before any ticket work begins. The
  script is graph-auth-check.ps1 in the powershell/ folder.

  Real incident: April 24 2026 -- user walked up for a BitLocker key. /pim was
  invoked mid-interaction and triggered an unexpected device code prompt, delaying
  the response. Morning pre-check was added the same day to prevent recurrence.

  Real use: Before looking up a device in Azure or doing an Exchange trace, Claude
  reminds you to activate the required PIM role. One command in the terminal,
  then back to the ticket.

  14. GHD END OF SHIFT DIGEST
--------------------------------------------------------------------------------

The morning digest has a companion: at the end of the shift Brent says
"give me my end of day digest" and Claude runs three JQL queries in parallel,
checks any pending Slack thread watches from memory, and returns a formatted
plain-text shift summary copied to clipboard.

THREE SECTIONS -- CLEANLY NON-OVERLAPPING:

  CLOSED TODAY
    Tickets resolved or closed during the shift. Shows the key resolution
    detail from the last comment and the time-to-resolution.
    Example from April 23 2026: 15 tickets closed in a single shift.

  PROGRESSED TODAY
    Open tickets that were updated, commented on, or moved forward during
    the shift but not yet closed. Shows current status and last action taken.

  NEW ASSIGNMENTS
    Tickets that came in today and are still open -- the incoming queue for
    the next morning's digest to pick up.

SHIFT TOTALS:
  A summary line at the bottom: X closed | X progressed | X new in | X total actioned.
  "Total actioned" = closed + progressed (new assignments came in, not actioned yet).

PENDING FOLLOW-UP INTEGRATION:
  Before pulling Jira data, Claude checks memory for any Slack threads flagged
  for monitoring (e.g., "waiting for Brandon's confirmation on GHD-96998").
  Those threads are read and the result is surfaced at the top of the digest --
  so nothing falls through the cracks between sessions.

  Real example from April 23 2026: The Vaisala viewLinc device ticket
  (GHD-96998) had been pending Brandon Fuller's (Netops) confirmation for two
  days. The EOD digest opened with: "Brandon replied: I believe its good to go."

  - Morning call volume from Shipping before most IT staff are even at their desks
  - P1 incidents that require parallel tracking of 3+ Slack threads at once

Practically speaking: two people, one environment spanning five continents.
Claude's value here is not novelty. It is speed, accuracy, and institutional memory --
the things a team this size cannot maintain manually at this volume.
